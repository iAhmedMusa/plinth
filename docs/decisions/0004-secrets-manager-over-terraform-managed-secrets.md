# 0004. Secrets Manager over Terraform-managed secrets

## Context

The RDS master password has to exist somewhere. Terraform variables,
plan output, and state are all places a plaintext secret can end up
committed, logged, or cached — none of them are safe by default.

## Decision

`manage_master_user_password = true` in
`terraform/modules/rds/main.tf` tells RDS to generate the master
password itself and store it directly in AWS Secrets Manager. There is
deliberately no `db_password` variable anywhere in `terraform/` —
Terraform never reads or writes the plaintext, so it cannot appear in a
`.tfvars` file, a `plan` diff, or state. Paired with `prevent_destroy`
on the RDS instance (and the EKS cluster) so an unreviewed `apply`
can't force-replace the resource the secret is scoped to.

## Alternatives considered

- **`random_password` resource + write to SSM Parameter Store from
  Terraform.** Gives more control — a custom rotation Lambda, a
  Terraform-visible password for one-off debugging. Rejected as the
  default because the password briefly exists in Terraform's own plan
  and state at creation time regardless of where it's written
  afterward, and every future rotation would require a coordinated
  Terraform-plus-SSM update rather than an independent, AWS-managed
  schedule. (This alternative is still documented as an option in
  `terraform/README.md` section 7 for teams that specifically need
  Terraform-visible rotation.)

## Consequences

Rotation is entirely AWS-managed and Terraform-independent — a real
operational win. The cost is that Terraform can never assert or verify
the password's value, and the initial password is generated by AWS
rather than chosen at provisioning time. Neither has any practical
impact here since nothing depends on a specific default password.
Application pods still need a separate mechanism (External Secrets
Operator or the Secrets Store CSI driver, under an IRSA-scoped role —
see ADR-0002) to read the real value at runtime, since Terraform never
holds it to hand off.
